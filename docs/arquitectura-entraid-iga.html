1 · Estado actual — Problemas identificados

🚨
Problema crítico: Licencias nunca se revocan

El equipo de mesa de ayuda asigna licencias M365 manualmente. Al no existir proceso de baja, las licencias permanecen activas aunque el usuario ya no sea estudiante, docente ni trabajador activo. Esto genera costo innecesario y riesgo de seguridad.

📋
Registro manual de emails
Mesa de ayuda
  • La mesa de ayuda crea/modifica cuentas directamente en Entra ID
  • No hay correlación automática con los datos de LAMB (Oracle)
  • 73K cuentas con atributos vacíos (givenName, surname, dept...)
  • Sin estándar de UPN — casos inconsistentes
💳
Licencias sin control de ciclo de vida
Sin IGA
  • Asignación manual → ningún proceso de revocación
  • Egresados siguen con licencia A1/A3 activa
  • Personal que renunció conserva acceso M365
  • No hay auditoría de quién asignó qué licencia ni cuándo
🔐
Sin gestión de permisos Entra ID
Riesgo seguridad
  • La mesa de ayuda tiene permisos excesivos en Entra ID
  • No hay separación entre operaciones de identidad y de licencias
  • MidPoint no tiene User.ReadWrite.All → no puede corregir esto aún
  • Sin Administrative Units por campus/unidad

Caso dual-afiliación: estudiante + trabajador

Si un usuario tiene simultáneamente BR-Estudiante y BR-Personal-General, MidPoint le asignará ambos AR. Entra ID group-based licensing asigna la licencia más alta disponible. 6 · Flujo de trabajo — Mesa de Ayuda

❌ Modelo actual (problemático)
  • Mesa de ayuda entra a Entra ID Admin Center
  • Busca al usuario por nombre o código
  • Crea o edita la cuenta manualmente
  • Asigna licencia manualmente (selecciona SKU)
  • No hay proceso de revocación → licencia queda para siempre
  • Sin auditoría de quién hizo qué cambio
  • Sin correlación con los datos de LAMB/Oracle
  • Errores frecuentes: UPN mal escrito, licencia incorrecta
✅ Modelo nuevo (con MidPoint IGA)
  • Mesa de ayuda entra a MidPoint Admin UI
  • Busca al usuario por personalNumber, DNI o nombre
  • Verifica que el usuario ya existe (importado de LAMB)
  • Asigna el Business Role correcto (ej: BR-Personal-General)
  • MidPoint provisiona automáticamente: cuenta Entra ID + grupo de licencia
  • Al remover el rol: MidPoint revoca licencia automáticamente
  • Toda acción queda en el audit log de MidPoint
  • Sin errores de UPN: se genera automáticamente desde LAMB
🛠
¿Qué hace la Mesa de Ayuda en MidPoint?
TareaCómo en MidPoint
Asignar acceso M365Agregar Business Role → auto-provisiona
Revocar acceso M365Quitar Business Role → auto-revoca licencia
Resetear contraseñaMidPoint → Credentials → Reset password (sincroniza a Entra ID)
Verificar estado cuentaMidPoint → User → Projections → Entra ID shadow
Reportar accesos activosMidPoint → Reports → Assignments
🚫
¿Qué NO debe hacer la Mesa de Ayuda?
  • Crear/editar cuentas directamente en Entra ID Admin Center
  • Asignar licencias directamente en Entra ID (sin pasar por MidPoint)
  • Cambiar UPN o email directamente en Entra ID
  • Agregar usuarios a grupos de licencia manualmente
  • Modificar membresías de grupos de seguridad
⚠

Los cambios directos en Entra ID serán sobrescritos en el siguiente recompute de MidPoint. El modelo IGA requiere que MidPoint sea la única fuente de verdad operacional.

10 · Diseño técnico MidPoint — Entitlements de grupos

🔧
Patrón canónico: kind=entitlement, intent=group + <association>

La membresía de grupos en Graph API NO se escribe como atributo outbound (ri:memberOfGroup es read-only). Se modela como association en el schemaHandling del resource. MidPoint llama POST /groups/{id}/members/$ref al agregar y DELETE /groups/{id}/members/{userId}/$ref al quitar.

Resource — 2 objectTypes requeridos

/* UPEU-EntraID-Graph.xml — agregar post-B3 */

/* objectType 1 (ya existe): account/default */
/* Agregar <association> en este objectType: */
<association>
  <ref>ri:group</ref>
  <kind>entitlement</kind>
  <intent>group</intent>
  <direction>objectToSubject</direction>
  <associationAttribute>ri:members</associationAttribute>
  <valueAttribute>ri:id</valueAttribute>
  <shortcutAssociationAttribute>
    ri:memberOfGroup
  </shortcutAssociationAttribute>
  <shortcutValueAttribute>icfs:uid</shortcutValueAttribute>
</association>

/* objectType 2: entitlement/group (nuevo) */
<objectType>
  <kind>entitlement</kind>
  <intent>group</intent>
  <default>false</default>
  <delineation>
    <objectClass>ri:GroupObjectClass</objectClass>
  </delineation>
  /* DTI crea los grupos — MidPoint solo gestiona members */
</objectType>

Construction en AR-M365-Student-A1

/* AR-M365-Student-A1.xml — diseño objetivo post-B3 */
<inducement>
  <construction>
    <resourceRef oid="2f11c057-..."/>
    <kind>account</kind>
    <intent>default</intent>
    <association>
      <ref>ri:group</ref>
      <outbound>
        <expression>
          <associationTargetSearch>
            <filter>
              /* nombre exacto del grupo en Entra ID */
              <q:text>
                attributes/icfs:name = "lic-m365-a1-students"
              </q:text>
            </filter>
            <searchStrategy>
              onResourceIfNeeded
            </searchStrategy>
          </associationTargetSearch>
        </expression>
      </outbound>
    </association>
  </construction>
</inducement>
✅
El lifecycle de grupos es completamente automático

Cuando MidPoint remueve un BR, las constructions del AR dejan de aplicar. MidPoint detecta la diferencia y llama la API de Graph para quitar al usuario del grupo → Entra ID revoca la licencia. Sin mappings especiales.
