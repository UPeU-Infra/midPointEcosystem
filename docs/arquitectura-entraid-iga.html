2 · Arquitectura objetivo — 2 mundos paralelos bajo un solo decisor

ℹ
MidPoint es el único decisor. Entra ID y OpenLDAP son destinos paralelos, no conectados entre sí.

Oracle LAMB es la fuente de verdad. MidPoint lee LAMB, aplica Business Roles → Application Roles, y escribe en DOS sistemas simultáneamente: Entra ID (mundo Microsoft: licencias, Teams, OneDrive, apps cloud) y OpenLDAP (mundo on-premise: Keycloak, RADIUS, apps académicas). Keycloak autentica contra OpenLDAP via User Federation — no contra Entra ID. Crear grupos manualmente en Entra ID o LDAP, o asignar licencias usuario por usuario, rompe la cadena IGA (ISO 24760-2 §6.4).

⚠
Keycloak NO autentica contra Entra ID

El flujo correcto es: usuario abre app → Keycloak → OpenLDAP (User Federation) → usuario autenticado. Entra ID es un sistema TARGET paralelo para el mundo Microsoft, no un proveedor de identidad para Keycloak. Configurar Keycloak como Identity Broker contra Entra ID introduciria dependencia de disponibilidad cloud y rompe la arquitectura on-premise.

4 · Grupos de seguridad y licencias M365

✅
Estrategia: Group-Based Licensing — grupos estáticos gestionados por MidPoint

NO usar grupos dinámicos de Entra ID. Para IGA correcto: grupos estáticos donde MidPoint agrega/quita miembros explícitamente via Graph API. Entra ID asigna y revoca SKUs automáticamente según membresía.

⚠
Los grupos mostrados abajo son diseño propuesto — aún no existen en el tenant

Los SKUs (STANDARDWOFFPACK_STUDENT, M365EDU_A3_FACULTY, etc.) sí son reales y están contratados por UPeU. Los grupos de seguridad (lic-m365-a1-students, etc.) se crearán cuando MidPoint tenga GroupMember.ReadWrite.All (B3).

⚠
Dual-afiliacion: estudiante + trabajador

Si un usuario tiene BR-Estudiante y BR-Personal-General simultaneamente, MidPoint le asigna ambos AR. Group-based licensing de Entra ID otorga la licencia mas alta disponible. Para evitar redundancia, el AR de menor privilegio debe omitir la construction de grupo cuando ya existe el de mayor privilegio.

🚨
Problema detectado en el tenant: más usuarios asignados que licencias contratadas

UPeU tiene contratadas N licencias de un tipo. Si hay más usuarios asignados que N, el exceso queda en estado inválido — esos usuarios pueden perder acceso sin previo aviso.

  • M365EDU_A5_FACULTY: contratadas 15, asignadas a 16 usuarios → 1 usuario con acceso A5 en riesgo
  • Microsoft_365_Copilot_EDU: contratadas 0 (SKU vencido o cancelado), asignadas a 8 usuarios → 8 usuarios usando Copilot sin licencia vigente

7 · Ciclo de vida JML (Joiner · Mover · Leaver)

⚠
Diseño objetivo — no refleja el estado actual

Este flujo describe el ciclo de vida IGA canónico según estándares ISO 24760 y NIST 800-63, y es la propuesta a adoptar progresivamente conforme se habiliten los permisos necesarios (B3). Actualmente las cuentas Entra ID se crean por el sistema de admisiones de forma independiente, sin ciclo de vida automatizado. Los pasos marcados en azul (Provisioning / Licencia) requieren User.ReadWrite.All + GroupMember.ReadWrite.All que aún no están habilitados.

👤+
Joiner
PasoAccion
1 FuenteLAMB registra nuevo trabajador o alumno
2 SyncMidPoint importa → lifecycleState=active
3 PoliticaObject Template asigna BR segun primaryAffiliation
4 ProvisioningCuenta Entra ID ya existe (creada por sistemas actuales). MidPoint la correlaciona y puede sincronizar atributos (post-B3, por decision institucional)
5 LicenciaMidPoint agrega al grupo → Entra ID asigna SKU
🔄
Mover
PasoAccion
1 FuenteLAMB cambia afiliacion (ej: alumno → egresado)
2 SyncMidPoint detecta cambio en primaryAffiliation
3 PoliticaBR anterior se remueve, nuevo BR se asigna
4 LicenciaSale del grupo de licencia anterior → Entra ID revoca SKU. Destino de licencia para egresados pendiente de decisión.

Leaver
PasoAccion
1 FuenteLAMB marca baja o fin de contrato
2 SyncMidPoint pierde afiliaciones activas

4 AccesoaccountEnabled=false → sin acceso M365

9 · Fases de implementacion

ℹ
Principio rector: potenciar lo que existe, no reemplazarlo

MidPoint no crea ni elimina cuentas Entra ID en esta etapa — las cuentas ya existen y seguirán siendo creadas por los sistemas actuales (admisiones, DTI). El objetivo es establecer gobierno de identidad progresivo: primero observar y correlacionar, luego sincronizar atributos, luego gestionar accesos. Los cambios estructurales (ciclo de vida completo, creación automática, revocación de licencias) son propuestas a evaluar y adoptar por decisión institucional.
