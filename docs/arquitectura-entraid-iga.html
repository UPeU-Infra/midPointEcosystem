💡
Principio rector: MidPoint es la única fuente de decisión de identidad

Dual-afiliacion: estudiante + trabajador

Si un usuario tiene BR-Estudiante y BR-Personal-General simultaneamente, MidPoint le asigna ambos AR. Group-based licensing de Entra ID otorga la licencia mas alta disponible. Para evitar redundancia, el AR de menor privilegio debe omitir la construction de grupo cuando ya existe el de mayor privilegio. Se maneja con condicion en el inducement del AR-M365-Student-A1 (activeShadowExists en el recurso EntraID con kind=account y AR-M365-Staff-A3).

6 · Flujo de trabajo — Mesa de Ayuda

Modelo actual — problematico
  • Mesa entra a Entra ID Admin Center directamente
  • Crea o edita cuentas y asigna licencias manualmente
  • Sin proceso de revocacion — licencias quedan indefinidamente
  • Sin auditoria de quién hizo qué cambio
  • Errores frecuentes: UPN mal escrito, licencia incorrecta, sin correlacion con LAMB
Modelo nuevo — con MidPoint IGA
  • Mesa opera exclusivamente en MidPoint Admin UI
  • Busca al usuario por personalNumber, DNI o nombre
  • Asigna Business Role correcto → MidPoint provisiona Entra ID automaticamente
  • Al remover el rol: licencia revocada automaticamente
  • Toda accion queda en audit log de MidPoint
  • UPN generado automaticamente desde LAMB — sin errores manuales

10 · Diseno tecnico MidPoint — Entitlements de grupos

🔧
Patron canonico: kind=entitlement, intent=group + <association>

La membresia de grupos en Graph API NO se escribe como atributo outbound (ri:memberOfGroup es read-only). Se modela como association en el schemaHandling. MidPoint llama POST /groups/{id}/members/$ref al agregar y DELETE /groups/{id}/members/{userId}/$ref al quitar.

Resource — 2 objectTypes requeridos

/* UPEU-EntraID-Graph.xml — agregar post-B3 */

/* objectType 1 (ya existe): account/default */
/* Agregar <association> en este objectType: */
<association>
  <ref>ri:group</ref>
  <kind>entitlement</kind>
  <intent>group</intent>
  <direction>objectToSubject</direction>
  <associationAttribute>ri:members</associationAttribute>
  <valueAttribute>ri:id</valueAttribute>
  <shortcutAssociationAttribute>
    ri:memberOfGroup
  </shortcutAssociationAttribute>
  <shortcutValueAttribute>icfs:uid</shortcutValueAttribute>
</association>

/* objectType 2: entitlement/group (nuevo) */
<objectType>
  <kind>entitlement</kind>
  <intent>group</intent>
  <default>false</default>
  <delineation>
    <objectClass>ri:GroupObjectClass</objectClass>
  </delineation>
  /* DTI crea grupos — MidPoint solo gestiona members */
</objectType>

Construction en AR-M365-Student-A1

/* AR-M365-Student-A1.xml — diseno objetivo post-B3 */
<inducement>
  <construction>
    <resourceRef oid="2f11c057-..."/>
    <kind>account</kind>
    <intent>default</intent>
    <association>
      <ref>ri:group</ref>
      <outbound>
        <expression>
          <associationTargetSearch>
            <filter>
              /* nombre exacto del grupo en Entra ID */
              <q:text>
                attributes/icfs:name = "lic-m365-a1-students"
              </q:text>
            </filter>
            <searchStrategy>
              onResourceIfNeeded
            </searchStrategy>
          </associationTargetSearch>
        </expression>
      </outbound>
    </association>
  </construction>
</inducement>

Al remover un BR, las constructions dejan de aplicar. MidPoint detecta la diferencia y llama Graph para quitar al usuario del grupo → Entra ID revoca el SKU. Sin mappings adicionales.
